The Main Correction

A streamed runtime is not yet a full agent loop.

The runtime you have now still does one main pass:

  1. read the latest query
  2. choose tools
  3. run tools
  4. summarize results

A Pi-inspired agent adds one more layer above this: it decides whether to keep going.

The Three Missing Behaviors

1. Real Loop

The agent needs a bounded loop such as:

while step < maxSteps:
  decide next action
  maybe call tool
  inspect result
  decide continue or stop

2. Real Decision Shape

The agent needs a first-class internal decision object, for example:

type AgentDecision =
  | { kind: "call_tool"; toolName: string; args: unknown }
  | { kind: "summarize" }
  | { kind: "stop"; reason: string };

3. Real Stop Rules

The loop needs boundaries:

  • maxSteps
  • stop on repeated tool calls
  • stop on empty or low-signal results
  • stop when there is no better next action

What Pi-Inspired Means Here

For this workspace, “Pi-inspired” should mean:

  1. the runtime is explicit and inspectable
  2. tool calls are structured
  3. the run can continue across multiple steps
  4. state and context are persisted in markdown files
  5. the run is reviewed by a small eval layer

It does not mean “copy an external Pi package exactly.” It means keeping the same operational ideas in your own codebase.

    tool-selection-rules.md

These four memory types serve different jobs:

  • working memory: current run state
  • episodic memory: what happened in one run
  • semantic memory: durable facts learned over time
  • procedural memory: how the agent should operate

The Context-Doc Plan

  SYSTEM.md
  MISSION.md
  TOOL_GUIDE.md
  EVAL_GUIDE.md
  RANKING_GUIDE.md

These files are stable inputs to each run. They are not memories. They are the durable instructions and reference documents the runtime loads before it starts.
