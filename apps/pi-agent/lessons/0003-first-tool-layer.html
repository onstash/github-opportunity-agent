Lesson 3

Add the First Tool Layer

Now we connect the runtime to the real world. The win in this lesson is not full GitHub integration yet. It is learning how two different tools can feed one shared opportunity pipeline.

The Goal

Your first useful agent needs exactly two domain tools:

  1. search_oss to find repositories or issues worth contributing to
  2. search_jobs to find job-like opportunities connected to GitHub organizations or projects

Both tools can return different raw shapes. That is normal. The important move is what happens next: normalize them into one shared Opportunity type.

What Each Tool Should Return First

Keep the first version intentionally narrow. Do not try to return every field GitHub exposes.

type RawOssHit = {
  repoName: string;
  repoUrl: string;
  issueTitle?: string;
  labels: string[];
  stars: number;
  updatedAt: string;
  topics: string[];
};

type RawJobHit = {
  orgName: string;
  postingUrl: string;
  roleTitle: string;
  summary: string;
  skills: string[];
  updatedAt: string;
  source: string;
};

The Crucial Translation Step

These raw types are not what the ranker should consume. Instead:

function normalizeOss(hit: RawOssHit): Opportunity
function normalizeJob(hit: RawJobHit): Opportunity

The normalizers are where you decide how domain-specific facts become shared ranking signals.

Example Mapping

OSS Mapping

  • title: issue title or repo name
  • organization: repo owner
  • topics: repo topics plus labels
  • recencyScore: based on updatedAt
  • effortScore: lower effort for beginner-friendly labels

Job Mapping

  • title: role title
  • organization: org or company name
  • topics: required skills
  • recencyScore: based on posting freshness
  • confidenceScore: higher for official org-owned sources

Minimal TypeScript Shape

type Opportunity = {
  kind: "oss" | "job";
  title: string;
  url: string;
  organization: string;
  summary: string;
  topics: string[];
  relevanceScore: number;
  recencyScore: number;
  effortScore: number;
  confidenceScore: number;
};

Once both tools emit this shape, your ranker becomes dramatically simpler because it no longer cares where the item came from.

What The Pi Tools Should Know

The model does not need direct GitHub API knowledge. It only needs tool descriptions clear enough to decide when to call each tool.

{
  name: "search_oss",
  description: "Find GitHub repositories and contribution issues relevant to a user profile"
}

{
  name: "search_jobs",
  description: "Find GitHub-adjacent job opportunities relevant to a user profile"
}

That separation is healthy: the model chooses, the tool fetches, the normalizer translates, the ranker compares.
