What The Current Runtime Still Hides

Today, your runtime does make decisions, but those decisions are implicit:

if no untried tool:
  stop
if next tool is search_oss:
  run search_oss
if next tool is search_jobs:
  run search_jobs

That works, but it hides agent choice inside control flow.

The Main Idea

Separate:

  1. choice — what should happen next?
  2. execution — now do that thing

This is the first explicit decision protocol.

The First Useful Shape

type AgentDecision =
  | {
      kind: "call_tool";
      toolName: "search_oss" | "search_jobs";
      query: string;
    }
  | {
      kind: "stop";
      reason: StopReason;
    };

This is intentionally small. Do not add summarize or memory-writing decisions yet.

How The Loop Changes

while iteration < maxIterations:
  decision = decideNextAction(state)

  if decision.kind == "stop":
    stop runtime

  if decision.kind == "call_tool":
    execute tool
    inspect result
    continue

The loop becomes easier to read because the runtime says what it is about to do before doing it.

What `decideNextAction(...)` Reads

  • current query
  • candidate tools
  • already executed tools
  • current iteration
  • stop rules

What It Returns

  • call_tool when a next action exists
  • stop when no safe or useful next action exists

Why This Is Better

  • It makes the runtime explainable.
  • It creates a clear place for eval later.
  • It creates a clear place for memory reads later.
  • It prevents loop logic from becoming a tangle of ad hoc branches.

What Not To Do Yet

  • Do not add model-generated decisions yet.
  • Do not add many decision kinds yet.
  • Do not bury stop-rule logic in three different helpers.
  • Do not skip the decision object just because there are only two tools today.
