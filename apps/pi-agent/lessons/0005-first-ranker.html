Lesson 5

Build the First Ranker

Now we make decisions. The first ranker should be simple, inspectable, and explainable. We are not chasing perfect intelligence yet. We are building a scoring model you can debug.

The Ranker’s Job

The ranker takes a list of normalized Opportunity items and answers two questions:

  1. Which opportunities fit this user best?
  2. Why did those opportunities rise to the top?

Start With Four Signals

type Opportunity = {
  kind: "oss" | "job";
  title: string;
  url: string;
  organization: string;
  summary: string;
  topics: string[];
  relevanceScore: number;
  recencyScore: number;
  effortScore: number;
  confidenceScore: number;
};

These are enough for a first pass because they capture fit, freshness, approachability, and trust.

Use A Weighted Sum First

function totalScore(opportunity: Opportunity): number {
  return (
    opportunity.relevanceScore * 0.45 +
    opportunity.recencyScore * 0.20 +
    opportunity.effortScore * 0.20 +
    opportunity.confidenceScore * 0.15
  );
}

Why this shape? Because relevance should dominate early. An irrelevant opportunity should not win just because it is fresh.

Keep Explanations Beside Scores

type RankedOpportunity = Opportunity & {
  totalScore: number;
  reasons: string[];
};
function explain(opportunity: Opportunity): string[] {
  const reasons: string[] = [];

  if (opportunity.relevanceScore >= 8) reasons.push("strong skill match");
  if (opportunity.recencyScore >= 7) reasons.push("fresh opportunity");
  if (opportunity.effortScore >= 7) reasons.push("approachable next step");
  if (opportunity.confidenceScore >= 7) reasons.push("credible source");

  return reasons;
}

This keeps the final CLI output legible and trustworthy.

Minimal Ranker Skeleton

function rank(opportunities: Opportunity[]): RankedOpportunity[] {
  return opportunities
    .map((opportunity) => ({
      ...opportunity,
      totalScore: totalScore(opportunity),
      reasons: explain(opportunity),
    }))
    .sort((a, b) => b.totalScore - a.totalScore);
}

That is enough for the first working CLI.

What Not To Do Yet

  • Do not add dozens of weights.
  • Do not bury logic inside prompts.
  • Do not mix ranking with fetching or normalization.
  • Do not optimize for jobs and OSS with completely separate formulas yet.

Retrieval Practice

  1. Why should relevance have the highest weight in the first model?
  2. Why should reasons be stored beside the score?
  3. Why is a weighted sum a good starting point?
