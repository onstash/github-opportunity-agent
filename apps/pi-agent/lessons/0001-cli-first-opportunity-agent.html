Lesson 1

Shape the CLI-First Opportunity Agent

Your first win is not “build the whole agent.” It is choosing the smallest architecture that can honestly work for both open source and jobs without collapsing into a pile of special cases.

Mission Tie-In

You want one TypeScript agent, built on Pi AI, that can inspect GitHub-shaped signals and answer a practical question: what should I pursue next, this OSS opportunity or this job opportunity?

That means our first design choice is architectural, not cosmetic.

The Key Idea

Do not build two separate agents first. Build one agent with two source adapters and one shared ranking shape.

CLI command
  -> build user profile
  -> ask Pi model to choose tools
  -> fetch OSS and job candidates
  -> normalize into one opportunity type
  -> rank
  -> explain the top picks

This lesson’s recommendation is partly an engineering inference from your mission. Pi AI gives us the model, context, tools, and event stream; the domain architecture is our layer on top of that.

What Pi AI Gives Us

  • A serializable Context with prompt, messages, and tools
  • Typed tool definitions using TypeBox schemas
  • models.stream(...) for event-by-event output
  • Provider flexibility if you switch models later

Source: Pi AI README.

What GitHub Gives Us

  • GraphQL for precise, flexible queries
  • REST endpoints for issues and search-oriented flows
  • A consistent way to gather repos, issues, and org signals
  • A foundation for later adapters outside GitHub

Sources: GitHub GraphQL API docs, GitHub REST issues docs.

The Minimal Data Model

Everything becomes easier once both opportunity types fit one envelope:

type Opportunity = {
  kind: "oss" | "job";
  title: string;
  url: string;
  organization: string;
  summary: string;
  topics: string[];
  relevanceScore: number;
  recencyScore: number;
  effortScore: number;
  confidenceScore: number;
};

If a source cannot be normalized into this shape, it is too early to add it.

Your First CLI Should Do Only This

  1. Accept a small profile: skills, interests, target roles.
  2. Fetch a few OSS candidates and a few job candidates.
  3. Print the top five with a one-line explanation each.
  4. Show stream events so the runtime is inspectable.

That is enough to be “working.” It does not need web UI, persistence, or autonomous loops yet.
