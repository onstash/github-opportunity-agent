Lesson 4

Write the Normalization Functions

This is where the agent stops being a pile of source-specific payloads and becomes a system with one clean comparison shape.

The Job Of A Normalizer

A normalizer translates raw source facts into a shared Opportunity object. It should be boring, predictable, and free of model logic.

type Opportunity = {
  kind: "oss" | "job";
  title: string;
  url: string;
  organization: string;
  summary: string;
  topics: string[];
  relevanceScore: number;
  recencyScore: number;
  effortScore: number;
  confidenceScore: number;
};

What Goes In

type RawOssHit = {
  repoName: string;
  repoUrl: string;
  repoOwner: string;
  issueTitle?: string;
  issueBody?: string;
  labels: string[];
  stars: number;
  updatedAt: string;
  topics: string[];
};

type RawJobHit = {
  orgName: string;
  postingUrl: string;
  roleTitle: string;
  summary: string;
  skills: string[];
  updatedAt: string;
  source: "official_org" | "repo_issue" | "discussion";
};

What The OSS Normalizer Should Do

function normalizeOss(hit: RawOssHit): Opportunity {
  return {
    kind: "oss",
    title: hit.issueTitle ?? hit.repoName,
    url: hit.repoUrl,
    organization: hit.repoOwner,
    summary: hit.issueBody ?? `Contribute to ${hit.repoName}`,
    topics: [...hit.topics, ...hit.labels],
    relevanceScore: 0,
    recencyScore: scoreRecency(hit.updatedAt),
    effortScore: scoreOssEffort(hit.labels),
    confidenceScore: scoreOssConfidence(hit.stars),
  };
}

relevanceScore can start as a placeholder if user-specific ranking comes later. That is fine. The important thing is to make the shape complete and consistent.

What The Job Normalizer Should Do

function normalizeJob(hit: RawJobHit): Opportunity {
  return {
    kind: "job",
    title: hit.roleTitle,
    url: hit.postingUrl,
    organization: hit.orgName,
    summary: hit.summary,
    topics: hit.skills,
    relevanceScore: 0,
    recencyScore: scoreRecency(hit.updatedAt),
    effortScore: scoreJobEffort(hit.summary),
    confidenceScore: scoreJobConfidence(hit.source),
  };
}

The two functions are allowed to compute signals differently. They only need to produce values that mean roughly the same thing at the output boundary.

Helper Functions Are Healthy

Keep source-specific heuristics in small helpers.

function scoreRecency(updatedAt: string): number
function scoreOssEffort(labels: string[]): number
function scoreOssConfidence(stars: number): number
function scoreJobEffort(summary: string): number
function scoreJobConfidence(source: RawJobHit["source"]): number

This keeps each normalizer readable and testable.

What To Avoid

  • Do not compute final personalized ranking here.
  • Do not mix tool fetching and normalization in one function.
  • Do not keep dozens of optional fields in the shared type on day one.
  • Do not let missing raw fields leak as undefined all over the output.

Retrieval Practice

  1. Why can both normalizers calculate signals differently and still be valid?
  2. Why is relevanceScore allowed to start as a placeholder?
  3. Why should helper functions hold source-specific heuristics?
