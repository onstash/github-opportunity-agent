Lesson 6

Lay Out the CLI App

You now understand the moving parts. This lesson gives each part a home so the first implementation can grow without turning into one giant file.

The Goal

The first CLI implementation should have clear boundaries between:

  1. Pi runtime setup
  2. tool definitions
  3. source fetching
  4. normalization
  5. ranking
  6. terminal rendering

Recommended File Layout

src/
  cli.ts
  models.ts
  types.ts
  tools.ts
  sources/
    oss.ts
    jobs.ts
  normalize/
    oss.ts
    jobs.ts
  ranking/
    rank.ts
  render/
    terminal.ts

This is small enough to start, but each layer still has its own responsibility.

What Each File Owns

Runtime Side

  • cli.ts: parse inputs, assemble context, run the flow
  • models.ts: create Pi models collection and select model
  • tools.ts: declare search_oss and search_jobs
  • render/terminal.ts: print stream events and final results

Domain Side

  • types.ts: shared app types like Opportunity
  • sources/oss.ts: raw GitHub OSS fetching
  • sources/jobs.ts: raw GitHub job-like fetching
  • normalize/*: source-specific translation
  • ranking/rank.ts: weighted scoring and reasons

The End-To-End Flow

cli.ts
  -> models.ts
  -> tools.ts
  -> sources/oss.ts and sources/jobs.ts
  -> normalize/oss.ts and normalize/jobs.ts
  -> ranking/rank.ts
  -> render/terminal.ts

The most important thing here is that each step hands off cleanly to the next.

What Not To Do

  • Do not put GitHub fetch code inside tools.ts and ranking in the same file.
  • Do not make cli.ts own all the business logic.
  • Do not let terminal rendering decide ranking behavior.
  • Do not duplicate shared types across files.

Smallest Useful Starting Point

If you need to collapse this even further for the first commit, keep these five:

src/
  cli.ts
  types.ts
  tools.ts
  normalize.ts
  rank.ts

Then split later when the first end-to-end path works.

Retrieval Practice

  1. Why should tools.ts not own ranking logic?
  2. Why should terminal rendering stay separate from ranking?
  3. Which file should own the shared Opportunity type?
